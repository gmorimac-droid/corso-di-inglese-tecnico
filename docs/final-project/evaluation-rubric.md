# Evaluation Rubric

## Purpose

This rubric is designed to support fair and transparent evaluation of the final project.

The project should not be judged only on grammar or vocabulary accuracy. A strong final project should also demonstrate:
- documentation logic;
- technical usefulness;
- structural coherence;
- terminological control;
- publication readiness.

## At a Glance

The final project is evaluated across six areas:

1. scope and coherence;
2. technical English quality;
3. terminology control;
4. documentation structure;
5. applied value and usability;
6. publication readiness.

## Performance Levels

| Level | General Meaning |
|---|---|
| Excellent | strong and consistent performance; very limited revision needed |
| Good | solid performance; minor weaknesses present |
| Satisfactory | acceptable but uneven performance; further revision needed |
| Weak | insufficient control; major improvement required |

## Rubric

### 1. Scope and Coherence

| Level | Descriptor |
|---|---|
| Excellent | The project has a clear purpose, a defined audience, and a coherent structure. Pages work together as one documentation product. |
| Good | The purpose is generally clear and most pages support it. Minor inconsistencies are present, but the repository remains credible. |
| Satisfactory | The purpose is partly clear, but scope control is uneven. Some pages feel disconnected, repetitive, or underdeveloped. |
| Weak | The repository lacks a stable objective. Content feels fragmented, generic, or loosely assembled. |

### 2. Technical English Quality

| Level | Descriptor |
|---|---|
| Excellent | Language is clear, professional, and well controlled. Grammar and sentence structure support technical communication effectively. |
| Good | Language is generally appropriate and understandable. Minor grammar or style issues are present, but usability remains good. |
| Satisfactory | Language is understandable but uneven. Weaknesses in grammar, phrasing, or register reduce professional credibility. |
| Weak | Language is frequently vague, inaccurate, informal, or structurally weak. The text does not reliably support technical communication. |

### 3. Terminology Control

| Level | Descriptor |
|---|---|
| Excellent | Terminology is accurate, consistent, and contextually appropriate. Key terms are used with discipline and stability. |
| Good | Most terminology is relevant and correctly used. Minor inconsistencies do not significantly affect overall quality. |
| Satisfactory | Terminology is partly controlled, but usage is sometimes generic, repetitive, or imprecise. |
| Weak | Terminology is inconsistent, weak, or overly general. The register does not feel professionally credible. |

### 4. Documentation Structure

| Level | Descriptor |
|---|---|
| Excellent | Pages are logically organised, easy to navigate, and internally well structured. Headings and links support the reader effectively. |
| Good | The repository is readable and mostly well structured. Some pages could be improved, but the overall hierarchy works. |
| Satisfactory | Basic structure is present, but organisation is uneven. Some headings, sections, or page flows are unclear. |
| Weak | The repository is poorly structured. Navigation, headings, and hierarchy do not support effective use. |

### 5. Applied Value and Usability

| Level | Descriptor |
|---|---|
| Excellent | The project contains clearly useful content such as templates, phrasebanks, examples, exercises, or guidance pages that can be applied directly. |
| Good | The project contains useful applied material, although some parts remain more descriptive than practical. |
| Satisfactory | Some useful content is present, but application value is uneven or limited. |
| Weak | The repository provides little practical value. Content remains abstract, generic, or non-operational. |

### 6. Publication Readiness

| Level | Descriptor |
|---|---|
| Excellent | The project is ready for publication with minimal or no further revision. Formatting, naming, and Markdown use are well controlled. |
| Good | The project is close to publication readiness. Only minor adjustments are needed. |
| Satisfactory | The repository is functional, but still needs visible revision before publication. |
| Weak | The repository is not ready for publication. Major work is still required. |

## Suggested Weighting

| Area | Weight |
|---|---:|
| Scope and coherence | 15% |
| Technical English quality | 25% |
| Terminology control | 15% |
| Documentation structure | 15% |
| Applied value and usability | 15% |
| Publication readiness | 15% |

## Reviewer Comment Prompts

Reviewers are encouraged to comment on at least these points:

- strongest aspect of the project;
- main issue affecting professional credibility;
- terminology recommendation;
- structural recommendation;
- priority action before publication.

## Self-Assessment Questions

Before final submission, the learner should ask:

- Does the repository feel coherent?
- Does it contain reusable professional content?
- Are the key terms stable across pages?
- Is the English controlled enough for technical readers?
- Would another reader understand how to use this site?

## Final Interpretation

A strong final project is not only linguistically acceptable.

It should also be:
- coherent;
- useful;
- reviewable;
- documentation-oriented;
- ready for structured publication.
