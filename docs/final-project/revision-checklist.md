?# Revision Checklist

## Purpose

Use this checklist to review the final project before submission or publication.

The goal is to verify that the repository is:
- coherent;
- readable;
- technically credible;
- structurally consistent;
- ready for publication.

## At a Glance

Review the project across six areas:

1. project-level coherence;
2. page-level quality;
3. language quality;
4. terminology control;
5. documentation usability;
6. publication readiness.

---

## 1. Project-Level Coherence

### Check
- The project has a clear objective.
- The target reader is identifiable.
- The repository feels like one documentation product.
- The main sections support the same purpose.
- The scope is realistic and controlled.
- Titles, sections, and examples match the declared topic.

### Warning Signs
- Some pages feel unrelated.
- The tone changes too much from page to page.
- Section names and actual content do not match.
- The repository mixes notes, exercises, and documentation without clear structure.

---

## 2. Page-Level Quality

### Check
- The title reflects the real content.
- The opening section explains the page purpose.
- Headings are meaningful.
- Paragraphs have a clear function.
- The page is not empty, repetitive, or generic.
- Examples are relevant and readable.
- The page adds real value to the repository.

### Warning Signs
- The page contains filler language.
- The same idea appears several times in slightly different wording.
- The content is too generic to be useful.
- Headings exist but do not help navigation.

---

## 3. Language Quality

### Check
- Sentences are grammatically controlled.
- The tone is professional rather than conversational.
- Vague adjectives are limited or explained.
- Active and passive voice are used intentionally.
- Long sentences remain readable.
- Punctuation supports clarity.
- Italian-influenced constructions have been corrected.

### Typical Problems to Remove
- *the system is composed by* → use *the system consists of*
- *it allows to monitor* → use *it allows users to monitor* or *it enables monitoring*
- *in according to* → use *in accordance with*
- *very important* without explanation
- *good performance* without measurable or contextual support

---

## 4. Terminology Control

### Check
- Key terms are used consistently.
- Abbreviations are introduced clearly.
- Similar terms are not treated as if they were identical.
- Glossary entries match actual page usage.
- The same component or process is not renamed without reason.

### Review Questions
- Do I use *fault*, *failure*, *issue*, and *anomaly* accurately?
- Do I distinguish *requirement* from *recommendation*?
- Do I use *system*, *subsystem*, *module*, and *component* consistently?
- Do I keep the same naming convention across pages?

---

## 5. Documentation Usability

### Check
- The reader can find information quickly.
- Section relationships are clear.
- Templates, examples, and phrasebanks are actually reusable.
- Theory and application are clearly separated.
- Internal links support navigation.

### Review Questions
- Are important pages linked from other relevant pages?
- Does the homepage explain how to use the site?
- Are templates really reusable?
- Are examples representative and useful?

---

## 6. Publication Readiness

### Check
- `mkdocs.yml` matches the real repository structure.
- File names are stable and consistent.
- Navigation entries match real files.
- The site renders correctly in local preview.
- There are no visible placeholder notes.
- Markdown formatting is clean.
- README or release notes match the current repository state.

---

## Fast Final Review

A practical final review can be done in this order:

1. read the homepage;
2. read the navigation from top to bottom;
3. open five representative pages from different sections;
4. review one glossary page, one module page, one exercise page, one template page, and one final-project page;
5. check whether the repository still feels coherent as a whole.

## Sign-Off Questions

The project is ready only if you can answer **yes** to these questions:

- Does the site behave like a documentation product?
- Would a new reader understand its purpose?
- Is the English suitable for technical readers?
- Is the terminology stable enough to support trust?
- Is the project ready to be shown publicly?

## Final Note

Revision is not a cosmetic stage.

In technical communication, revision is part of the product itself. A well-revised repository is easier to read, easier to trust, and easier to maintain.
