# Page Architecture Checklist

## Purpose

This checklist is used to review the internal structure of a page before publication. A strong page architecture improves readability, supports navigation, and reduces the probability of content duplication or conceptual drift.

The goal is not to force every page into the same rigid template. Instead, the goal is to ensure that each page has a visible logic and performs its intended function clearly.

## Structural Review Checklist

### Page identity
- Does the title clearly reflect the page topic?
- Is the scope of the page immediately understandable?
- Can the reader distinguish this page from related pages in the same section?

### Opening section
- Does the page explain its purpose near the beginning?
- Does the opening define the value of the page for the reader?
- Is the context clear enough for a first-time visitor?

### Internal sequencing
- Are sections arranged in a logical order?
- Does the page move from overview to detail, or from principle to application, in a controlled way?
- Are transitions clear?

### Functional completeness
- Does the page contain all expected elements for its type?
- If it is a glossary page, does it define terms and usage notes?
- If it is a module page, does it include outcomes, language focus, examples, and tasks?
- If it is a template page, does it provide reusable structure and guidance?

### Reader usability
- Can the reader scan the page efficiently?
- Are headings informative rather than generic?
- Is excessive textual density avoided through segmentation?
- Are lists, examples, and notes used where they improve comprehension?

### Repository integration
- Does the page link to relevant related pages?
- Is the terminology aligned with the appropriate glossaries?
- Does the page support the architecture of the repository instead of standing in isolation?

## Common Structural Failures

The following problems are common and should be corrected before release:

- title too broad or too vague;
- opening paragraph that does not define purpose;
- duplicated concepts spread across different headings;
- abrupt shifts from theory to exercises without transition;
- dense blocks of text with no structural support;
- a page that feels like notes rather than documentation.

## Minimum Viable Standard

Before a page is published, it should at least satisfy the following standard:

1. clear title;
2. clear purpose;
3. visible section logic;
4. coherent terminology;
5. at least minimal internal or cross-referenced usability.

## Recommended Reviewer Comment Frame

When a page fails architectural review, feedback should be specific. For example:

- "The page contains useful content, but the purpose is not defined early enough."
- "Sections 2 and 4 overlap and should be merged or differentiated."
- "The examples are relevant, but the reader lacks a clear progression from concept to application."
- "Cross-links to the glossary and template pages would improve usability."
