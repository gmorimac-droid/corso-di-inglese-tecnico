# Release Readiness Checklist

## Purpose

This checklist is intended for use before publishing a page set, section update, or repository release. It is the final quality gate between drafting and public availability.

Release readiness is broader than proofreading. A page can be grammatically correct and still fail release readiness because of weak navigation, inconsistent terminology, incomplete structure, or insufficient documentation value.

## Pre-Release Checklist

### Content quality
- The content is complete enough for publication.
- The page has a clear purpose and coherent scope.
- The wording is professional, readable, and consistent.
- Vague claims and unsupported evaluations have been removed.

### Terminology quality
- Key terms are aligned with the glossary pages.
- Requirement language is used correctly.
- No major terminology conflicts exist across related pages.

### Structural quality
- Headings are clear and informative.
- The page architecture is coherent.
- Dense sections have been segmented where needed.
- Lists, examples, and notes support readability appropriately.

### Linking quality
- Relevant cross-links have been added.
- Broken or missing internal references have been checked.
- The page is integrated into the repository and not isolated.

### Publication quality
- File name and page title are appropriate.
- Markdown formatting is stable.
- The page works within the MkDocs navigation logic.
- Any local conventions used in the repository have been respected.

## Section-Level Release Questions

If an entire section is being released or revised, ask the following:

- Does the section have a visible internal logic?
- Do the pages within the section complement each other without unnecessary duplication?
- Is the section mature enough to represent the standard of the repository?
- Are high-value entry pages, index pages, and cross-links in place?

## Recommended Final Step

Before release, perform one last review from the perspective of a first-time visitor. Ask not only whether the content is correct, but whether it is genuinely usable, navigable, and worth consulting.

## Release Standard

A page should be considered release-ready only when it is:

- technically credible;
- linguistically stable;
- structurally coherent;
- terminologically governed;
- integrated into the repository.
