# Email Writing Exercises

## Purpose

Technical emails are often underestimated. In reality, they are one of the most frequent forms of professional communication in engineering environments. A technical email must be clear, appropriately formal, concise, and operationally useful. It should support action, not create ambiguity.

This page develops technical email writing for status updates, clarification requests, anomaly reporting, coordination, and document review.

---

## Exercise 1 — Subject Line Control

Write an appropriate email subject line for each situation:
1. request for confirmation of test configuration;
2. transmission of revised installation procedure;
3. anomaly observed during vibration testing;
4. request for missing interface data;
5. scheduling of maintenance review meeting.

### Reviewer Focus
A good subject line is specific, short, and traceable.

---

## Exercise 2 — Clarification Request

Write an email requesting clarification on the following issue:
- interface specification revision number unclear;
- two versions in circulation;
- installation team requires confirmation before proceeding;
- schedule impact possible if clarification is delayed.

### Output Requirements
Include:
- clear purpose;
- concise context;
- explicit request;
- professional closing.

---

## Exercise 3 — Reporting an Anomaly by Email

Draft an email to report the following:
- intermittent communication loss observed during testing;
- preliminary checks completed;
- no thermal anomaly;
- possible connector issue;
- additional inspection planned.

### Target Tone
Professional, factual, non-alarmist.

---

## Exercise 4 — Follow-Up After Meeting

Write an email summarizing the outcome of a short technical meeting.

### Meeting Outcomes
- revised bracket design to be reviewed;
- alignment issue to be verified during next build;
- maintenance team to provide feedback on access constraints;
- updated drawing expected by Friday.

### Output Goal
The email should function as both summary and action tracker.

---

## Exercise 5 — Sending a Revised Document

Write an email accompanying the transmission of a revised technical procedure.

### Required Information
- document title;
- revision identifier;
- main change introduced;
- request for review;
- suggested response deadline.

---

## Exercise 6 — Improving Weak Email Writing

Rewrite the email below.

> Hi,  
> We found some problems with the unit. Please check the document because maybe the values are not correct. Let me know soon.  
> Thanks.

### Tasks
1. identify the weaknesses;
2. rewrite the email in professional technical English;
3. improve subject line, context, and request clarity.

---

## Exercise 7 — Managing Tone

Write two versions of the same message:
- version A for an internal engineering colleague;
- version B for a more formal external stakeholder.

### Scenario
You need to request confirmation of environmental test limits before finalizing the test plan.

### Reflection
Explain what changes between the two versions.

---

## Exercise 8 — Escalation Without Overstatement

Write an escalation email based on the following:
- unresolved interface ambiguity;
- installation team blocked;
- schedule pressure increasing;
- immediate clarification requested;
- no criticism of individuals.

### Reviewer Focus
The email should remain factual, professional, and solution-oriented.

---

## Exercise 9 — Response to a Request

Write a reply email confirming that:
- the revised drawing has been reviewed;
- no issue was found with the new cable routing;
- one note is added regarding minimum bending radius;
- updated comment sheet attached.

### Output Requirement
The email should be concise but complete.

---

## Exercise 10 — Repository-Ready Email Model

Write a polished email model that could later be reused in the repository as a template. Choose one scenario:
- request for technical data;
- document review request;
- anomaly notification;
- action follow-up after technical meeting.

### Length
140 to 220 words.
