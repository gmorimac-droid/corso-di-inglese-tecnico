?# Briefing Template

## Purpose

This template supports short technical briefings delivered in professional contexts such as design reviews, maintenance coordination, operational updates, project status communication, or risk escalation meetings.

A briefing differs from a report because it is designed for spoken delivery, often supported by slides or a short note. It must therefore be structured for clarity, progression, and audience retention.

A good technical briefing should answer:

- what the situation is;
- why it matters;
- what the current status is;
- what the risks or constraints are;
- what decisions or actions are required.

---

## Recommended Oral Flow

1. Opening and purpose
2. Technical context
3. Current status or issue
4. Key findings or implications
5. Required actions or decision points
6. Closure and next steps

---

## Template

```md
# Technical Briefing – [Title]

## 1. Purpose
[State the purpose of the briefing in one or two sentences.]

## 2. Background
[Provide the minimum context required by the audience.]

## 3. Current Status
[Summarize the current technical or documentary status.]

## 4. Key Points
- [Key point 1]
- [Key point 2]
- [Key point 3]

## 5. Risks / Constraints / Dependencies
- [Risk or limitation]
- [Dependency]
- [Assumption]

## 6. Recommended Actions
- [Action / decision required]
- [Action / decision required]

## 7. Closing Statement
[Provide a short conclusion and indicate the expected next step.]
```

---

## Model Example

# Technical Briefing – Sensor Enclosure Revision Status

## 1. Purpose
The purpose of this briefing is to summarize the current status of sensor enclosure revision C and identify the actions required before formal release.

## 2. Background
Revision C was introduced to improve cable routing clarity, simplify maintenance access, and align the enclosure design with updated installation constraints.

## 3. Current Status
The revised configuration is technically acceptable at preliminary review level. However, two documentation points remain open and should be resolved before release.

## 4. Key Points
- Internal routing clarity has improved compared with the previous baseline.  
- Alignment tolerance is referenced descriptively but not yet stated as a formal requirement.  
- The environmental paragraph still requires normalization against the current installation profile.

## 5. Risks / Constraints / Dependencies
- Release quality may be reduced if requirement wording remains incomplete.  
- Downstream documentation may inherit ambiguity if the environmental assumptions are not corrected.  
- Closure depends on coordinated input from systems and technical documentation.

## 6. Recommended Actions
- Add explicit alignment tolerance to the requirement section.  
- Update the environmental paragraph before document release.  
- Conduct a short closure review after revision issue update.

## 7. Closing Statement
In summary, the revision is close to release readiness, but targeted wording improvements are still required to ensure documentary robustness and integration clarity.

---

## Delivery Notes

- Use short spoken units, not long written paragraphs.
- Signal transitions clearly: “first”, “second”, “finally”, “as a result”.
- Distinguish fact from recommendation.
- End with a decision or action point.
- Keep the closing sentence explicit.
