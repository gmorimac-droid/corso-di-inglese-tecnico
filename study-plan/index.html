    Study Plan¶

    How the Course Can Be Delivered¶

    This course can be delivered in different formats depending on the learner profile, institutional constraints, and expected final output.

    The most suitable models are shown below.

    At a Glance¶

    Model Typical Duration Best Use Case
    Extended semester model 18–22 weeks academic delivery with steady progression
    Intensive professional model 6 blocks over several weeks workplace upskilling and compact delivery
    Annual blended model long-duration format portfolio development and integration with other technical work

    Delivery Models¶

    Extended Semester Model¶

    • 18 modules over 18 to 22 teaching weeks;
    • 1 core module per week;
    • regular revision sessions;
    • final project started progressively from around week 5.

    This model is suitable for learners who need time to build terminology, writing habits, and documentation discipline gradually.

    Intensive Professional Model¶

    • 18 modules grouped into 6 blocks;
    • 2 or 3 sessions per week;
    • stronger focus on applied writing and documentation output;
    • suitable for professional training and workplace language development.

    This model works well when the course is linked to operational documentation needs.

    Annual Blended Model¶

    • slower rhythm;
    • more time for revision and reworking;
    • stronger integration with project work, internships, or technical subjects;
    • ideal when the MkDocs site is expected to become a substantial portfolio.

    This model is particularly effective when the learner is building not only language competence, but also a reusable documentation asset.

    Suggested Course Progression¶

    The course is divided into six phases.

    Phase 1 — Foundations¶

    Modules 01–04

    Main focus: - shared vocabulary; - grammar control; - technical reading strategies; - basic documentary logic.

    Expected output: - short glossary entries; - structured notes; - simple technical descriptions.

    Phase 2 — Description and Explanation¶

    Modules 05–08

    Main focus: - describing components and assemblies; - explaining systems and interfaces; - presenting workflows and sequences; - reporting performance and test outcomes.

    Expected output: - system descriptions; - process summaries; - short evaluation notes.

    Phase 3 — Professional Writing¶

    Modules 09–12

    Main focus: - technical writing principles; - emails and professional messages; - reports and summaries; - requirements and compliance language.

    Expected output: - email drafts; - report sections; - requirement statements; - structured documentation pages.

    Phase 4 — Operational and Risk-Related Communication¶

    Modules 13–15

    Main focus: - maintenance language; - troubleshooting and fault analysis; - safety and incident communication.

    Expected output: - maintenance notes; - troubleshooting summaries; - safety statements and incident-related writing.

    Phase 5 — Institutional and Sector-Specific Language¶

    Modules 16–17

    Main focus: - defence-oriented technical English; - procurement and programme language; - tone, neutrality, and audience awareness.

    Expected output: - sector-adapted technical wording; - neutral professional documentation; - controlled terminology in formal contexts.

    Phase 6 — Documentation Publication¶

    Module 18 + Final Project

    Main focus: - repository structure; - Markdown organisation; - MkDocs logic; - GitHub-based publication workflow.

    Expected output: - structured documentation site; - reusable project assets; - final learner portfolio.

    Recommended Weekly Routine¶

    A practical weekly pattern is:

    Day 1 — Input¶

    • read the module;
    • identify key concepts;
    • extract new terminology.

    Day 2 — Language Focus¶

    • review grammar and phrasebanks;
    • observe patterns and collocations;
    • compare weak and strong wording.

    Day 3 — Applied Task¶

    • complete a writing, reading, or speaking activity;
    • produce a first draft.

    Day 4 — Revision and Documentation¶

    • revise the draft;
    • update the glossary;
    • convert useful content into Markdown.

    Day 5 — Review¶

    • conduct self-review, peer review, or instructor review;
    • improve the output before storing it in the repository.

    Minimum Evidence to Collect¶

    During the course, the learner should maintain a small but consistent set of working assets:

    • a personal glossary;
    • a language issues log;
    • a revision checklist;
    • at least one refined output per module;
    • a mini portfolio of model paragraphs, summaries, and reports.

    These assets make the final project easier to assemble and improve.

    Practical Recommendation¶

    The final project should not be left until the end of the course.

    A better approach is to build the documentation set progressively: - revise each useful task; - turn good outputs into repository pages; - link them to glossaries, templates, and modules; - improve structure over time.

    This turns the final project into a process of integration rather than a last-minute compilation task.
