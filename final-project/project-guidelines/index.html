    Project Guidelines¶

    General Instruction¶

    The final project must be developed as a coherent documentation repository. The learner should treat the repository as a professional technical product built for a clear audience and a clear use case.

    The project is not expected to reproduce the complexity of a full industrial documentation package. However, it should demonstrate a credible understanding of how technical information is structured, controlled, and presented in English.

    Project Objective¶

    Your objective is to produce a MkDocs-based documentation site that shows your ability to:

    • write professional technical English at B2 level;
    • organise information logically across multiple pages;
    • use terminology consistently;
    • support the reader through examples, explanations, and reusable structures;
    • build a repository that is publishable, navigable, and reviewable.

    Mandatory Requirements¶

    Your project should satisfy the following requirements.

    Content Requirements¶

    The repository must contain:

    • a clear homepage or introduction page;
    • a project scope explanation;
    • multiple technical content pages with meaningful body text;
    • at least one glossary component;
    • at least one document template or model;
    • at least one applied or exercise-oriented page;
    • a portfolio or reflective page documenting your contribution and development.

    Language Requirements¶

    Your writing should:

    • remain technically clear;
    • avoid uncontrolled informal language;
    • distinguish between requirement, recommendation, description, and observation;
    • use consistent terminology;
    • avoid unnecessary translation-like phrasing from Italian;
    • demonstrate control of functional grammar used in technical communication.

    Editorial Requirements¶

    Your documentation should:

    • use consistent page titles;
    • apply headings logically;
    • avoid empty sections;
    • present information in a reusable way;
    • include internal links where useful;
    • maintain a stable structure across similar pages.

    Technical Repository Requirements¶

    Your repository should:

    • use a clear docs/ structure;
    • include an operational mkdocs.yml file;
    • respect naming conventions for pages and folders;
    • use Markdown consistently;
    • be ready for local preview and GitHub-based publication.

    Recommended Content Model¶

    A strong project often follows a content model such as this:

    1. Introduction layer
      Explains what the repository is, who it is for, and how to use it.

    2. Knowledge layer
      Provides concepts, definitions, and technical explanations.

    3. Application layer
      Provides exercises, models, templates, or scenarios.

    4. Review layer
      Provides checklists, correction notes, or quality expectations.

    5. Portfolio layer
      Explains how the project evolved and what was learned.

    Topic Selection Guidance¶

    Choose a project topic that is specific enough to remain coherent, but broad enough to generate several useful pages. Suitable themes include:

    • technical English for maintenance teams;
    • terminology and phraseology for engineering trainees;
    • documentation for a generic monitoring system;
    • safety and compliance communication in technical environments;
    • civil versus defence-oriented technical terminology;
    • troubleshooting and reporting language;
    • system description and test-report writing.

    Avoid topics that are too broad, too vague, or disconnected from the course objectives.

    Page Quality Expectations¶

    Each page should contain more than surface-level statements. Strong pages generally include:

    • a defined scope;
    • clear headings;
    • technical terminology with contextual use;
    • practical examples;
    • distinctions between strong and weak phrasing where relevant;
    • cross-reference to related pages or concepts.

    Weak pages usually contain generic filler language, undefined claims, or excessive repetition.

    Writing Guidance¶

    When drafting your pages, apply the following principles:

    Write for use¶

    The page should help a reader understand, compare, apply, or produce technical language.

    Prefer precision over decoration¶

    Professional documentation should be informative, not ornamental.

    Keep paragraphs purposeful¶

    Each paragraph should perform a function.

    Control terminology¶

    If you use a term such as integration, compliance, fault, or supportability, make sure the context supports the meaning.

    Make the reader's task easier¶

    Structure content so the reader can find information quickly.

    Evidence to Include¶

    You should try to include evidence of the following skills:

    • system description;
    • process explanation;
    • technical comparison;
    • requirement-style writing;
    • report-style writing;
    • corrective revision;
    • glossary development;
    • professional tone management.

    Submission Readiness Criteria¶

    Before considering the project ready, verify the following:

    • all key pages contain meaningful content;
    • links are valid and useful;
    • terminology is coherent across sections;
    • section titles match actual content;
    • obvious grammar and wording issues have been corrected;
    • the repository can be read as a documentation product, not a draft notebook.

    Recommended Review Sequence¶

    1. Review project scope.
    2. Review page map.
    3. Review terminology consistency.
    4. Review language quality.
    5. Review navigation and cross-links.
    6. Review formatting and Markdown consistency.
    7. Review publication readiness.

    Final Advice¶

    Treat the final project as an engineering communication artefact. Even when the topic is educational, the repository should reflect discipline, traceability, structure, and user awareness.
