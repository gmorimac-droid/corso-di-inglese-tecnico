    Project Guidelines¶

    What You Are Expected to Build¶

    The final project should be developed as a coherent documentation repository.

    The goal is not to produce a large quantity of pages. The goal is to build a small but credible MkDocs site that shows your ability to use B2-level technical English in a structured, professional, and reviewable way.

    At a Glance¶

    Your project should demonstrate that you can:

    • organise information across multiple pages;
    • write in clear and controlled technical English;
    • use terminology consistently;
    • support the reader through examples and structured content;
    • build a site that is navigable, reusable, and publication-ready.

    Mandatory Requirements¶

    Your project should include:

    • a clear homepage or introduction page;
    • a page explaining project scope;
    • several substantial content pages;
    • at least one glossary component;
    • at least one template or model document;
    • at least one applied or exercise-oriented page;
    • a portfolio or reflection page.

    Language Requirements¶

    Your writing should:

    • remain clear and professional;
    • avoid uncontrolled informal language;
    • distinguish between description, requirement, recommendation, and observation;
    • use terminology consistently;
    • avoid obvious translation-like phrasing;
    • show control of technical grammar and document-oriented wording.

    Editorial Requirements¶

    Your pages should:

    • use clear and consistent titles;
    • apply headings logically;
    • avoid empty or weak sections;
    • present information in a reusable way;
    • include internal links where useful;
    • maintain a stable structure across similar pages.

    Technical Repository Requirements¶

    Your repository should:

    • use a clear docs/ structure;
    • include a working mkdocs.yml;
    • follow stable naming conventions;
    • use Markdown consistently;
    • be ready for local preview and GitHub-based publication.

    Recommended Content Model¶

    A strong project usually includes five layers:

    1. Introduction Layer¶

    Explains what the repository is, who it is for, and how it should be used.

    2. Knowledge Layer¶

    Provides definitions, concepts, explanations, and technical reference content.

    3. Application Layer¶

    Provides exercises, templates, models, or practical scenarios.

    4. Review Layer¶

    Provides checklists, quality guidance, or revision support.

    5. Portfolio Layer¶

    Explains what was developed, improved, and learned during the project.

    Choosing a Good Topic¶

    Choose a topic that is: - specific enough to remain coherent; - broad enough to support several useful pages; - clearly connected to the course objectives.

    Suitable themes may include:

    • technical English for maintenance teams;
    • terminology and phraseology for engineering trainees;
    • documentation for a generic monitoring system;
    • safety and compliance communication;
    • civil vs defence-oriented terminology;
    • troubleshooting and reporting language;
    • system description and test-report writing.

    Avoid topics that are too broad, too vague, or disconnected from the course focus.

    What Makes a Page Strong¶

    A strong page usually includes:

    • a clear purpose;
    • a defined scope;
    • useful headings;
    • technical terminology used in context;
    • practical examples;
    • meaningful links to related pages.

    A weak page usually contains:

    • generic filler text;
    • repeated ideas;
    • vague claims;
    • unclear structure;
    • little practical value.

    Writing Guidelines¶

    When drafting, apply these principles:

    Write for use¶

    The page should help the reader understand, compare, apply, or produce technical language.

    Prefer precision over decoration¶

    Professional documentation should be informative, not ornamental.

    Keep paragraphs purposeful¶

    Each paragraph should have a clear function.

    Control terminology¶

    Technical terms should be used consistently and in the correct context.

    Help the reader¶

    Structure content so that information can be found quickly.

    Skills the Project Should Show¶

    Try to include evidence of the following:

    • system description;
    • process explanation;
    • technical comparison;
    • requirement-style writing;
    • report-style writing;
    • revision and improvement;
    • glossary development;
    • professional tone control.

    Final Readiness Checklist¶

    Before considering the project ready, verify that:

    • all key pages contain meaningful content;
    • internal links work correctly;
    • terminology is coherent across sections;
    • headings match the real content;
    • obvious grammar and wording issues have been corrected;
    • the repository reads like documentation, not like draft notes.

    Recommended Review Sequence¶

    A practical review order is:

    1. review project scope;
    2. review page map;
    3. review terminology consistency;
    4. review language quality;
    5. review navigation and cross-links;
    6. review formatting and Markdown consistency;
    7. review publication readiness.

    Final Advice¶

    Treat the final project as a real engineering communication artefact.

    Even if the topic is educational, the repository should show: - structure; - clarity; - consistency; - reader awareness; - documentary discipline.
