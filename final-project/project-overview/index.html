    Final Project Overview¶

    What the Final Project Is¶

    The final project is the main consolidation task of the course.

    Its purpose is to show that the learner can use B2-level technical English in a structured, professional, and documentation-oriented way.

    Instead of producing a single essay or presentation, the learner builds a small technical documentation site using MkDocs and GitHub.

    At a Glance¶

    The final project should demonstrate that the learner can:

    • organise technical information clearly;
    • write in controlled professional English;
    • maintain terminological consistency across pages;
    • distinguish between different document functions;
    • build a small but coherent documentation repository.

    Project Format¶

    The final output should function as a mini documentation site, not as a collection of disconnected exercises.

    Possible project scenarios include:

    • a technical training documentation set;
    • an internal engineering knowledge base;
    • a system introduction and support handbook;
    • a maintenance and troubleshooting guide;
    • a safety and compliance documentation set;
    • a civil vs defence terminology reference;
    • a dual-use engineering communication repository.

    The scenario should remain realistic, clear, and professionally neutral.

    What a Good Project Should Show¶

    A good final project should show five things.

    1. Documentation Logic¶

    The repository should be structured as documentation, not as free-form prose.

    Pages should have: - a clear purpose; - a predictable structure; - a direct relationship to reader needs.

    2. Technical Language Control¶

    The English should reflect a usable B2 technical level.

    This means: - generally controlled grammar; - relevant terminology; - clear sentence structure; - limited vagueness; - consistent professional tone.

    3. Internal Coherence¶

    The repository should behave like one project.

    This includes: - consistent terminology; - stable naming conventions; - coherent page structure; - useful internal links.

    4. Usability¶

    The site should help the reader do something practical, such as: - understand a system; - review a process; - interpret a requirement; - use a template; - compare technical terminology.

    5. Publication Readiness¶

    The repository should be organised in a way that is suitable for MkDocs publication and GitHub-based versioning.

    Recommended Scope¶

    The final project does not need to be very large, but it should be coherent and complete enough to show planning, revision, and sustained writing effort.

    A balanced project usually includes:

    • overview pages;
    • technical description pages;
    • glossary content;
    • one or more support pages;
    • templates or model documents;
    • practice or portfolio material.

    Minimum Suggested Deliverables¶

    A solid minimum project may include:

    • 1 homepage or overview page;
    • 3 to 5 substantial content pages;
    • 1 glossary section with at least 20 useful terms;
    • 1 language-support or phrasebank page;
    • 2 practical templates;
    • 1 exercise or guided practice page;
    • 1 portfolio or reflection page.

    A stronger project will usually include better internal linking, more realistic examples, and more consistent editorial control.

    Define the Reader Before Writing¶

    Before drafting the project, define the main reader.

    Typical readers may include: - engineering trainees; - junior maintenance technicians; - documentation learners; - project support staff; - system-integration trainees; - professionals moving from Italian technical writing to English-language documentation.

    The site should be written for a reader who needs clarity, structure, and practical usefulness.

    What Counts as Evidence of Learning¶

    The final project should provide visible evidence in these areas:

    • terminology control
    • functional writing competence
    • documentation structure
    • revision and improvement
    • digital documentation literacy

    In practical terms, the project should show that the learner can write, revise, organise, and publish technical content in a controlled way.

    Simple Workflow¶

    A practical workflow is:

    1. define the project objective;
    2. identify the target reader;
    3. choose the main content categories;
    4. create the page map;
    5. draft the core pages;
    6. add glossary and support material;
    7. include templates or applied pages;
    8. review language quality;
    9. review structural coherence;
    10. prepare the site for publication.

    Common Risks¶

    Common project weaknesses include:

    • pages that are too generic;
    • inconsistent terminology;
    • poor distinction between lesson notes and documentation;
    • repetition across pages;
    • weak internal linking;
    • vague wording without technical support;
    • titles and headings that do not match the content clearly.

    These issues should be addressed during revision.

    Final Advice¶

    The best final projects are not always the largest.

    The strongest projects are usually the ones that are: - focused; - consistent; - well structured; - clear in purpose; - easy to review and navigate.
