    Final Project Overview¶

    Purpose of the Final Project¶

    The final project is the consolidation point of the course. It is designed to verify that the learner can apply Technical English at B2 level in a structured, professional, and documentation-oriented way.

    Unlike a traditional end-of-course assignment, this project is not limited to a single essay or presentation. It is a documentation package developed as a small but credible technical knowledge base and published through MkDocs and GitHub.

    The final product should demonstrate that the learner can:

    • organise technical information logically;
    • write in clear and controlled professional English;
    • maintain terminological consistency across multiple pages;
    • distinguish between descriptive, procedural, instructional, and evaluative writing;
    • produce documentation suitable for engineering, maintenance, systems, safety, or programme-support contexts;
    • manage a lightweight documentation repository using Markdown and version control.

    General Project Concept¶

    The learner is required to build a coherent mini-site that functions as a technical documentation environment rather than as a generic language exercise. The project may simulate one of the following scenarios:

    • a documentation package for a technical training course;
    • an internal engineering knowledge base;
    • a system introduction and support handbook;
    • a maintenance and troubleshooting guide;
    • a safety and compliance documentation set;
    • a civil/defence terminology reference environment;
    • a dual-use engineering communication repository.

    The chosen scenario should remain realistic and professionally neutral. The site should not read like a collection of disconnected exercises. It should read like a controlled documentation product built for a defined audience.

    Expected Project Characteristics¶

    The final project should meet the following characteristics:

    1. Documentation Logic¶

    The project must be structured as documentation, not as free-form prose. Pages must have a clear function, a predictable internal structure, and a direct relationship to user needs.

    2. Technical Language Control¶

    The English used in the repository should reflect B2-level technical competence. This means:

    • grammar is generally controlled;
    • terminology is relevant and stable;
    • sentences are clear and operational;
    • wording avoids unnecessary vagueness;
    • professional tone is maintained throughout.

    3. Internal Coherence¶

    The repository must behave like one project. Terms, naming conventions, headings, page structure, and cross-references should support a coherent reader experience.

    4. Usability¶

    The documentation should help the reader do something: understand a system, review a process, interpret a requirement, compare terminology, or use a template.

    5. Publication Readiness¶

    The project should be technically ready for publication on GitHub and presentation via MkDocs.

    Recommended Project Scope¶

    A reasonable final project should include a balanced combination of the following content categories:

    • overview pages;
    • technical description pages;
    • glossary pages;
    • practical exercises or guided tasks;
    • templates or model documents;
    • quality or review guidance;
    • project notes or learner portfolio material.

    The project does not need to be extremely large, but it must be sufficiently complete to demonstrate planning, consistency, and sustained writing ability.

    Minimum Suggested Deliverables¶

    A strong minimum project should include:

    • 1 homepage or course overview page;
    • 3 to 5 substantial module-like pages;
    • 1 glossary section with at least 20 useful terms;
    • 1 phrasebank or language support page;
    • 2 practical document templates;
    • 1 exercise or practice page;
    • 1 final reflection or portfolio page.

    A stronger project will include more cross-linking, more realistic examples, and clearer editorial governance.

    Target Reader¶

    Before writing, the learner should define a primary reader. Typical target readers include:

    • engineering trainees;
    • junior maintenance technicians;
    • technical documentation learners;
    • project support staff;
    • systems integration trainees;
    • personnel transitioning from Italian technical communication into English-medium documentation contexts.

    The repository should be written for a reader who needs practical clarity more than stylistic flair.

    Evidence of Learning¶

    The final project should provide observable evidence in at least five areas:

    1. Terminology control
      The learner selects and uses technical vocabulary appropriately.

    2. Functional writing competence
      The learner can write descriptions, instructions, notes, and summaries for different technical purposes.

    3. Documentation structure
      The learner can organise content through headings, sections, tables, lists, and linked pages.

    4. Review and revision
      The learner can improve wording, consistency, and clarity after drafting.

    5. Digital documentation literacy
      The learner can place content in a repository format suitable for versioning and publication.

    Suggested Workflow¶

    A recommended workflow is the following:

    1. Define the documentation objective.
    2. Identify the target reader.
    3. Select the main content categories.
    4. Build the page map.
    5. Draft the core pages.
    6. Populate glossaries and support pages.
    7. Add templates or applied materials.
    8. Review language quality.
    9. Review editorial coherence.
    10. Prepare the site for publication.

    Common Project Risks¶

    Typical weaknesses in final projects include:

    • pages that are too short and generic;
    • inconsistent terminology;
    • unclear distinction between course notes and technical documentation;
    • excessive repetition across sections;
    • weak internal linking;
    • vague wording such as good, efficient, easy, or robust without technical support;
    • poor alignment between title, section heading, and actual content.

    These risks should be addressed during the revision stage.

    Final Recommendation¶

    The best final projects are usually not the most ambitious in quantity. They are the most consistent in purpose, language, and structure. A focused repository with clear documentation logic is more credible than a large collection of loosely related pages.
