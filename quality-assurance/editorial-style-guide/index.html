    Editorial Style Guide¶

    Purpose¶

    This page defines the editorial conventions that should be applied across the repository. Its objective is not to impose unnecessary rigidity, but to ensure that the documentation remains readable, professional, coherent, and maintainable over time.

    In an engineering-oriented knowledge base, inconsistent style quickly becomes a functional problem. It can weaken readability, blur distinctions between requirements and explanations, create terminological drift, and reduce the repository's credibility.

    Core Editorial Principles¶

    1. Clarity before ornament¶

    Language should be precise and functional. Prefer wording that helps the reader understand a system, process, requirement, or communication pattern without interpretive effort.

    2. Consistency before variation¶

    A repository benefits more from stable formulation patterns than from stylistic variety. When a concept has already been defined and named, the same naming logic should be preserved unless there is a strong reason to revise it.

    3. Professional neutrality¶

    The tone should remain professional, informative, and controlled. Avoid conversational exaggeration, emotional emphasis, and promotional wording.

    4. Usability as an editorial criterion¶

    A technically correct paragraph is still weak if the reader cannot use it efficiently. Structure, headings, sequencing, scannability, and internal references are therefore part of editorial quality.

    Tone and Register¶

    Use a register that is:

    • technical but accessible;
    • formal but not bureaucratic;
    • authoritative but not inflated;
    • instructional where needed;
    • descriptive where appropriate;
    • evaluative only when supported by explicit criteria.

    Avoid¶

    • vague praise such as excellent, very good, highly impressive;
    • unsupported criticism such as poorly designed or weak without explanation;
    • colloquial transitions such as basically, kind of, a lot of;
    • rhetorical padding.

    Sentence Style¶

    Preferred features¶

    • moderate sentence length;
    • explicit subject and action;
    • clear causal or logical relations;
    • visible distinction between fact, recommendation, and requirement.

    Good example¶

    The revised interface reduces cable routing complexity and improves inspection access during routine maintenance.

    Weak example¶

    The new interface is much better and makes everything easier.

    The weak version is unacceptable because it is vague, non-technical, and unsupported.

    Paragraph Design¶

    Each paragraph should usually perform one primary function, such as:

    • introducing a concept;
    • defining a term;
    • explaining a process;
    • comparing two approaches;
    • describing a risk;
    • presenting a review criterion.

    Avoid paragraphs that mix too many functions without clear sequencing.

    Terminology Rules¶

    • Use the preferred term consistently once it has been established.
    • Avoid unnecessary synonyms for the same technical object.
    • Distinguish clearly between general-language approximations and technical terms.
    • Where a term may create confusion, define it explicitly on first use or link to the relevant glossary page.

    Requirement Language¶

    When writing specifications or controlled instructions, use modal language deliberately:

    • shall for formal requirements;
    • must for strict obligation or constraint, especially where safety or compliance is involved;
    • should for recommendation;
    • may for permission or acceptable option;
    • can for capability.

    These modal verbs should never be treated as interchangeable.

    Formatting Guidance¶

    • Use informative headings.
    • Prefer short bulleted lists when listing criteria, steps, or comparisons.
    • Use tables only when they improve comparison or navigation.
    • Keep emphasis limited. Bold should support navigation, not decoration.
    • Use examples deliberately and label them clearly.

    Reviewer Questions¶

    A reviewer applying this style guide should ask:

    • Is the page clear at first reading?
    • Does the wording remain professional throughout?
    • Are weak adjectives and vague claims avoided?
    • Does the page sound like part of the same repository as the other pages?
    • Would a technical learner or professional be able to reuse the wording?

    Final Rule¶

    The repository should read as if it were maintained by one disciplined editorial team, even when individual pages are drafted at different times or by different contributors.
